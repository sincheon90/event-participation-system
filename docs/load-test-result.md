# 負荷テスト結果

## 1. 概要

イベント開始直後に参加リクエストが集中する状況を想定し、イベント参加 API に対して 5 種類の負荷テストを実施した。

```http
POST /api/events/{eventId}/missions/{missionId}/participate
```

本書では JMeter の JTL および HTML Dashboard の集計値を基に、API の成功率、応答時間、Throughput、重複参加の遮断結果を評価する。テスト環境の詳細は [load-test-plan.md](load-test-plan.md) を参照する。

- 実施日: 2026-07-15（JST）
- 負荷生成ツール: Apache JMeter
- 対象 Event ID: `1`
- 対象 Mission ID: `1`

## 2. テスト条件

| シナリオ | Threads | Ramp-up | Loop Count | 総リクエスト数 |
|---|---:|---:|---:|---:|
| 通常負荷 | 100 | 30 秒 | 100 | 10,000 |
| 高負荷 | 250 | 10 秒 | 200 | 50,000 |
| スパイク | 500 | 1 秒 | 20 | 10,000 |
| 重複参加 | 1,000 | 1 秒 | 1 | 1,000 |
| 継続負荷 | 100 | 30 秒 | 1,000 | 100,000 |

各 JTL のデータ行数はそれぞれ 10,000、50,000、10,000、1,000、100,000 件であり、すべて計画したリクエスト数と一致した。

---

## 3. 集計方法

- 応答時間と Throughput は、各 JMeter Dashboard の `statistics.json` にある `Total` の値を使用した。
- 実行時間は、JTL の最初のリクエスト開始時刻から最後のレスポンス完了時刻までを算出した。
- JMeter の Throughput は Ramp-up を含むテスト全体を基準に算出される。
- HTTP `201 Created` を参加成功、`409 Conflict` を重複参加の遮断として集計した。
- 重複参加テストの `409 Conflict` は JMeter 上ではエラーになるが、本テストでは期待する業務結果として評価した。

## 5. シナリオ別評価

### 5.1 通常負荷

- 10,000 件すべてが `201 Created` で完了した。
- 平均応答時間は 9.36 ms、p99 は 45 ms で、通常負荷では安定した応答を確認した。
- 最大応答時間は 807 ms であり、少数の遅いリクエストが存在した。
- 30 秒の Ramp-up とほぼ同時にテストが終了したため、337.06 req/s は主に負荷上昇区間の測定値である。

### 5.2 高負荷

- 50,000 件すべてが成功し、エラーは発生しなかった。
- Throughput は 1,560.16 req/s まで上昇した。
- 平均応答時間は 118.19 ms、p95 は 371 ms、p99 は 638 ms だった。
- 成功率は維持したものの、通常負荷に比べてテールレイテンシが大きく増加した。

### 5.3 スパイク

- 500 Threads を 1 秒で投入しても、10,000 件すべてが成功した。
- 平均応答時間は 308.02 ms、p99 は 897.98 ms、最大は 1,537 ms まで上昇した。
- Throughput は 1,458.36 req/s で、高負荷テストの 1,560.16 req/s と同程度だった。
- この 2 シナリオの実行条件では、Threads の増加が処理量の増加につながらず、待ち時間として現れたことから、バースト時の処理待ちが発生したと考えられる。

### 5.4 重複参加

- 同一ユーザーによる 1,000 件のリクエストのうち、1 件だけが `201 Created` となった。
- 残り 999 件は `409 Conflict` で拒否され、API 応答上は重複参加を防止できた。
- JMeter のエラー率は 99.9% だが、これは期待した `409 Conflict` をエラーとして扱ったためであり、業務上の判定は成功である。
- API の `201 Created` 1 件と非同期ログの `SUCCESS` 1 件、`DUPLICATE` 999 件が一致しており、重複参加防止の結果に整合性があることを確認した。

### 5.5 継続負荷

- 100,000 件すべてが成功し、エラーは発生しなかった。
- 平均応答時間は 26.14 ms、p99 は 59 ms、最大は 199 ms だった。
- Throughput は 1,860.78 req/s で、今回のテスト中で最も高い値となった。
- 通常負荷と異なり、Ramp-up 後も各 Thread がリクエストを繰り返したため、全体 Throughput が高くなったと考えられる。
- 実行時間は約 53.74 秒であるため、長時間運転時のリソース枯渇や性能劣化を評価する耐久テストとしては十分ではない。

## 6. 非同期処理時間

### 6.1 測定方法

API の同期処理完了後に生成されたイベントの時刻と、Kafka Consumer が結果ログを保存する際の時刻との差を集計した。

```text
非同期処理時間(ms)
  = participation_result_logs.created_at
  - participation_result_logs.event_created_at
```

| カラム | 測定上の意味 |
|---|---|
| `event_created_at` | 同期参加処理の完了後、Kafka 送信用イベントを生成した時刻 |
| `created_at` | Consumer が結果ログを永続化する際の `@PrePersist` 実行時刻 |

この値には Producer 送信、Kafka Broker、Consumer 待機、および Consumer の結果ログ保存処理が含まれる。

API の JTL と非同期処理時間は、2026-07-15 の同一負荷テスト実行から取得した。シナリオごとに API 応答性能と非同期結果保存の状況を比較する。

### 6.2 集計結果

| シナリオ | 保存件数 | SUCCESS | DUPLICATE | 最小 | 平均 | p50 | p90 | p95 | p99 | 最大 |
|---|---:|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| 通常負荷 | 10,000 | 10,000 | 0 | 0.59 ms | 226.04 ms | 16.44 ms | 763.67 ms | 986.05 ms | 1,266.02 ms | 1,346.05 ms |
| 高負荷 | 50,000 | 50,000 | 0 | 19.52 ms | 15,435.81 ms | 17,847.27 ms | 20,149.03 ms | 20,242.98 ms | 20,512.68 ms | 20,641.11 ms |
| スパイク | 10,000 | 10,000 | 0 | 14.34 ms | 3,288.11 ms | 3,902.40 ms | 4,666.73 ms | 4,728.27 ms | 4,798.21 ms | 4,830.39 ms |
| 重複参加 | 1,000 | 1 | 999 | 39.84 ms | 855.87 ms | 841.76 ms | 986.76 ms | 1,002.39 ms | 1,029.56 ms | 1,034.73 ms |
| 継続負荷 | 100,000 | 100,000 | 0 | 0.87 ms | 23,999.55 ms | 28,889.39 ms | 31,503.29 ms | 31,817.56 ms | 32,159.73 ms | 32,204.51 ms |

### 6.3 評価

- 全シナリオで保存件数がリクエスト数と一致し、合計 171,000 件の参加結果ログが最終的に保存された。
- 通常負荷の p50 は 16.44 ms だったが、p99 は 1,266.02 ms であり、一部のイベントで処理待ちが発生した。
- 高負荷では平均 15.44 秒、p99 20.51 秒となり、Consumer の処理能力を超えるイベントが滞留したことが示唆される。
- スパイクでは平均 3.29 秒、p99 4.80 秒となり、瞬間的な投入による処理待ちが確認された。
- 重複参加の結果ログも 1,000 件すべて保存された。成功 1 件に加え、期待した重複結果 999 件も非同期記録の対象となっている。
- 継続負荷では平均 24.00 秒、p99 32.16 秒となり、今回の条件で最も大きな非同期処理遅延が発生した。
- API はリクエストを処理できていても、非同期側では最大約 32.20 秒の遅延が生じているため、Consumer lag の監視と Worker の処理能力評価が必要である。

## 7. 総合評価

### 確認できたこと

- 重複参加テストを除く 170,000 件は、HTTP レベルですべて正常に処理された。
- 500 Threads の急激な投入でも、タイムアウトや 5xx エラーは発生しなかった。
- 同一ユーザーからの 1,000 件の参加要求を、1 件の成功と 999 件の競合応答に制御できた。
- 100,000 件の連続リクエストを、エラーなし、p99 59 ms で処理した。
- 再実行した非同期処理測定では、全 171,000 件の参加結果ログが保存された。
- API のリクエスト件数と非同期結果ログの件数・status 分布が一致し、結果ログ保存までのデータ整合性を確認した。

### 性能上の注意点

- 高負荷の p99 は 638 ms、スパイクの p99 は 897.98 ms で 1 秒以内だったが、通常負荷と比べてテールレイテンシは増加した。
- 高負荷とスパイクでは Throughput が同程度で、スパイク時は応答時間だけが悪化した。
- 非同期処理は高負荷で p99 20.51 秒、継続負荷で p99 32.16 秒に達しており、API の成功率だけでは後続処理の健全性を判断できない。

## 8. 今後のテスト計画

### 8.1 Duration ベースの継続負荷テスト

- JMeter の Scheduler を使用し、Loop Count ではなく実行時間で制御する。
- 第 1 段階は 500 req/s を 30 分間維持し、合計 900,000 件を処理する。
- API の p95・p99、エラー率、JVM・MySQL・Kafka のメトリクス、Consumer lag が時間経過で悪化しないか確認する。

### 8.2 Worker 数による非同期処理能力の比較

- 高負荷シナリオの 50,000 件を固定し、Worker を 1、2、3 台に変更して同じテストを実行する。
- 非同期処理の平均、p95、p99、最大時間と、全結果ログの保存完了時間を比較する。
- Worker 数に対して処理能力がどこまで比例して向上するか確認する。

### 8.3 Redis・Kafka の有無による比較

- 現行構成（Redis + Kafka）を基準とする。
- Redis を使用せず DB Unique 制約だけで重複を防止する構成と比較し、API 応答時間、Throughput、DB 負荷を測定する。
- Kafka の非同期処理と、結果ログを同期保存する構成を比較し、API 応答時間と後続処理完了時間を測定する。

### 8.4 実運用に近い混合シナリオ

- 50,000 件のうち、新規参加 40,000 件（80%）、重複参加 10,000 件（20%）を混在させる。
- 250 Threads、Ramp-up 10 秒を基準とし、`201 Created` と `409 Conflict` が同時に発生する状況を再現する。
- 成功・重複の件数、API p95・p99、Redis と DB の負荷、非同期結果ログの整合性を確認する。

### 8.5 障害・回復テストと監視

- 負荷実行中に Redis または Kafka を停止・再起動し、API のエラー、データ欠損、重複処理、復旧時間を確認する。
- すべてのシナリオで CPU、メモリ、GC、DB Connection Pool、Kafka Consumer lag を同時に記録する。
- 各テスト終了後に `participations`、`participation_result_logs`、status 別件数を照合し、データ整合性を確認する。
